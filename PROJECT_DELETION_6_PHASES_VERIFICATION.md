## üìù ADDITIONAL NOTES

### Why localStorage Instead of Database?

**Decision**: Use client-side localStorage for deletion/creation state
**Reasoning**:
- ‚úÖ Faster (no network request)
- ‚úÖ Immediate UI updates
- ‚úÖ No database schema changes required
- ‚úÖ Works across all environments
- ‚úÖ Per-user state without auth complexity

**Trade-offs**:
- ‚ùå State lost if browser cache cleared
- ‚ùå Not synced across devices
- ‚ùå Not backed up automatically

**Acceptable because**: Projects are derived from conversations, not critical data. User can always recreate projects.

### Why Not Actually Delete Conversations?

**Decision**: Retain all conversations, just reassign them
**Reasoning**:
- ‚úÖ Data preservation (user's chat history is valuable)
- ‚úÖ Reversible (can restore project later)
- ‚úÖ Audit trail (can see what was in deleted project)
- ‚úÖ Search still works (content not lost)
